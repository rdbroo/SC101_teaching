What is R and RStudio?

RStudio is an integrated development environment (IDE) designed for working with R, a powerful programming language widely used for statistical computing, data analysis, and visualisation.

  • It’s open source, meaning it’s free!
  • Whether you are performing simple calculations or analysing complex social science datasets, RStudio streamlines the process and is widely used in academia and industry for its flexibility and open-source nature.

Why learn R?

  • Highly flexible: Allows advanced customisation.
  • Employability: Knowledge of R is a sought-after skill in research and industry.

CRAN is an essential resource for R users and developers, and it ensures that the R ecosystem is accessible, reliable, and well-maintained.

PART TWO: FAMILIARISING YOURSELF WITH RSTUDIO

RStudio Layout

When opening RStudio for the first time, you’ll see three main panes (see, Image 1):

  1. Console: Displays results and output from R code. More recent output will appear at the bottom. Old outputs will remain, unless you clear with the brush icon.
  2. Environment: Contains datasets, variables, and command history.
  3. Utilities: Displays files, visualisations, packages, and help documentation.

Where do you write your code?

You can type commands directly into the Console, but this approach has its limitations. The main drawback is that the Console does not allow you to save your work unless you manually copy and paste the commands, which can be time-consuming and inefficient.

Instead, use an R Script for better organisation and reproducibility.

To create a script:

  1. Navigate to File > New File > R Script.

When you open a script, it will appear in the top-left pane of RStudio. While you can have multiple scripts open at the same time, it’s usually best to focus on one to keep things organised. Each open script or dataset will appear as a tab in this pane, allowing you to easily switch between them as needed.

Structuring your script:

It’s essential to structure your scripts in a consistent and coherent way. This becomes especially important if you’re conducting data analysis for a publication, as it’s best practice to ensure your code is reproducible. Reproducibility means that another researcher, given access to your data, should be able to run your script and obtain the same results. This principle is a cornerstone of scientific integrity—and you’re a scientist, after all!

  • Add comments with # to explain your code. On a single line, anything that follows a # won’t run as code. You’ll notice that comments appear in green too, making them easy to identify. Without # R will interpret the text as code, resulting in an error message because it won’t understand what you mean.

When RStudio doesn’t understand. Try it yourself!

  1. Type the following directly into your R Script:
This is a comment, and it won't run as code
  1. Then, click the button. Alternatively, you can click CTRL+Enter.

You’ll get an error message in the Console.

  1. Now, try again, but this time add a # to the start of the line:
# This is a comment, and it won't run as code
  1. Again, click the button or press CTRL+Enter.

Hopefully, this time the error message doesn’t appear.

Understanding Console Output in RStudio

Ready State

The blue arrow in the RStudio console indicates that the console is ready for new commands. When you see this symbol, the previous code execution has finished successfully.

Busy State

If the blue arrow is not visible, the console is still busy running the current code.

During this time Monitor the execution progress. If necessary, interrupt the process by clicking the stop button in the RStudio toolbar.

The Order of the Console Output

New console output will appear at the bottom, keeping your workflow uninterrupted. Previous outputs will stay intact, providing a comprehensive view of your progress—unless you choose to clear the slate with a simple click of the brush icon .

Tips for Smooth Console Use

  • Keep an eye on the console output to understand what your code is doing.
  • Use comments (#) in your scripts to document the purpose of your commands, making debugging easier.
  • Clear the console as needed by using the brush icon for a cleaner workspace.

To make your scripts readable and clear, focus on maintaining a well-organised structure. A great place to start is by adding some key details at the beginning of your script. These details act as a “header” and provide important context for anyone reading or using your code.

Example of a script header:

# Author: Your Name
# Date: 2025-01-15
# Description: This script analyses survey data to identify trends in responses.

R Scripts should be set up in a sequential manner. So, the commands and functions that you write should not depend on code that doesn’t appear until later in the script.

PART THREE: ORGANISING AND SAVING FILES

Like word documents, scripts need to be saved. To do so, navigate to File > Save, and then find the folder where you would like to save. You should save your file regularly, as RStudio sometimes crashes!

We’ve already discussed the importance of keeping scripts well-organised and properly annotated with clear, legible notes. However, it’s equally important to maintain a coherent and organised folder structure for all your files.

Over time, you’ll likely accumulate a substantial collection of scripts, data files, and other project-related materials. Without a clear system, it can become challenging to locate a specific file-especially one you wrote three years ago. Imagine needing to revisit an old analysis or, as a researcher, having to defend your work to a sceptical peer questioning the integrity of your published results.

Best Practices for file naming and organisation

  1. Use clear and consistent file names (e.g., survey_analysis_2025.R).
  2. Maintain a logical folder structure, keeping related files grouped together:
    • Scripts
    • Data
    • Outputs

Consider creating folders for each Year of Study and Module. Within these, you can further categorise your materials using sub-folders for different components. For example, under a folder named Seminar and Lab Notes, you might create sub-folders for scripts, data and notes.

Now, Save your script!

Error and Warning Messages

Error messages in R appear in the Console in red and typically start with the word “Error”, followed by a brief description of the issue. While these messages can provide some insight into what went wrong, they may not always be immediately clear. For example, in the case above, the error indicates that the issue stems from the addition of an unexpected ] symbol.

If the message isn’t clear, don’t worry-it’s a normal part of coding! You can often resolve the problem by:

  1. Carefully rereading your code to check for typos or syntax errors.

  2. Searching the error message on Google or using generative AI tools to find explanations and solutions.

  3. Asking your class teacher, once you’ve exhausted the above.

With practice, you’ll get better at interpreting error messages and debugging your code.

Error messages are different from ‘warning messages’. These are alerts about potential issues in your code. Unlike errors, warnings do not stop the code from running; you will still get an output. However, the output might not be what you intended, so it’s worth reviewing the warning to ensure everything is working as expected.

PART FIVE: DESCRIPTIVE STATISTICS

Working with Datasets

In this course, and throughout your study, we’ll be using ‘datasets’ or ‘data frames’. As you’re aware from the lecture, a dataset is a structured collection of data. Typically, it is organised into rows and columns, that represents information about variables and observations. It is a fundamental unit of data analysis and is used to store, organise, and analyse data in a consistent format.

The Structure of a Dataset

Throughout this course (and across the methods modules in this department) we’ll primarily be using wide form datasets.

A wide-form dataset is a data structure in which each row typically represents an individual observation (e.g., a person, a study participant, a specific case), and the columns contain multiple variables for that observation.

In a wide-form dataset:

  • Rows: Represent individual observations or cases (e.g., people, places, events).

  • Columns: Represent variables, which are characteristics or attributes measured for each observation (e.g., age, income, weight).

  • Cells: Contain specific values for a given variable and observation.

What on Earth are Functions?

Good question. Take some time to read this section, as it’s really important!

Functions in R are reusable blocks of code designed to perform specific tasks. They take inputs (called arguments), process them, and return outputs. Functions simplify repetitive tasks and make your code more efficient.

head(), str(), data() are all functions.

For each function, the arguments (the inputs) are contained within the brackets.

So, in the previous code segment the function head() takes the dataset USArrests and presents the first few rows of the dataset USArrests. That’s what the head() function is designed to do.

There are many functions already in R (what we call, ‘base R’ functions), and you can create own functions too (but let’s not get ahead of ourselves). In a lot of cases you will need to download functions developed by other people. New functions are made by developers and contained in Packages.

Packages are collections of functions, datasets, and documentation bundled together to extend R’s capabilities. It’s why R is so darn good! Packages make R highly versatile for data analysis, visualisation, machine learning, and more. Packages are typically stored in the CRAN, but can also be found on Github and in other repositories.

To install packages (and hence get amazing new functions to do wild new things), we can run use the install.packages() function. I know, it’s a bit confusing, but we need functions to get functions.

So, to download a new package we would firstly run:

install.packages("package_name")

We swap out ‘package_name’ for the name of the package, of course. Note, you need to use the double quotation marks, as it won’t work without them.

Then, to use the package, we would use the function library(), with the package name contained within the brackets, like this:

library(package_name)

So, a complete example for the package ggplot ggplot would be:

install.package("ggplot2) # download the ggplot package
library(ggplot) # make it usable within this R session

Packages need updating periodically to ensure you have access to the latest features, bug fixed and updates. You can do this using the following code:

update.packages() # this will update all packages
install.packages("package_name") # change 'package_name' for the package name
Importing Data into RStudio

We’re not importing a dataset into R today. However, for real-world research, you wouldn’t use an ‘inbuilt’ dataset, but would instead import real data. The code for this depends on the type of data file you’ll use.

For CSV Files (files ending with ‘.csv’):

data <- read.csv("path/to/your/file.csv")

For Excel Files (.xlsx):

library(readxl)
data <- read_excel("path/to/your/file.xlsx", sheet = "Sheet1")

You’ll notice that we first download a package called readxl, this is because base R doesn’t allow us to important Excel files, so we have to download some new functions that are in a different package.

For Stata Files (.dta):

library(haven)
data <- read_dta("path/to/your/file.dta")

Again, we have to download a new package called haven to import .dta and .sav files.

For SPSS Files (.sav):

library(haven)
data <- read_sav("path/to/your/file.sav")

For R Data (.rds) Files:

data <- readRDS("path/to/your/file.rds")

You’ll need to adjust the file path and file name contained with the brackets. We also place each dataset into an data frame object called ‘data’. You could call it anything, but data is most intuitive. If you import it without creating a new data object (i.e., without including data <-), then the dataset will be imported, but will retain it’s original file name.

There are other data file types too, but these are the common ones. SPSS, STATA are alternative software programmes used for statistical analysis. They each have their own strengths and weaknesses, but we prefer RStudio!

Descriptive Statistics for ‘Arrests for Murder’

Say, as a criminologist we’re interested in arrests for murder across the United States in 1973. It would be useful to initially produce some summary statistics for this variable.

We can use the summary() function to produce some descriptive statistics (incl. the mean, median, range and the interquartile range) for this variable. The code will look like this:

For the variable Murder, we can use this code:

summary(USArrests$Murder)

You’ll notice here the inclusion of the $ sign. The sign is used to select a variable. We’re essentially saying apply the the summary() function to the Murder variable in the USArrests dataset. If we don’t specify the murder variable (with $Murder), summary statistics for all variables will be produced.

Note, R is case sensitive! Correctly capitalise Murder and USArrests, otherwise you’ll get an error message!

Using the summary() function for variable Murder should provide the following output:

We can alternatively produce the mean using the mean() function:

mean(data$USArrests)

The mean tells us that the number of arrests for murder in the United States in 1973 is 7.788 per 100,000 people on average. The range is indicated by the difference between the minimum and maximum number, so the range is between 0.8 and 17.4. The median is 7.25.

So, we’ve got some measures of central tendency (mean and median) and some measures of dispersion (Interquartile range and range). The interquartile range is the distance between the 1st quartile (4.075) and the 3rd quartile (11.250). The range is the distance between the minimum value (0.8) and the maximum value (17.4).

What about the mode? Well, there isn’t an inbuilt function to produce the mode, but there is one in the statip package, called mfv() (which standards for ‘most frequent value’). So, let’s install the statip package using this code:

install.packages("statip") # install package
library(statip) # lets make it usable in this R session

Now we can use the mfv() function to obtain the mode for number of arrests for murder:

mfv(USArrests$Murder)

This should produce the following output:

This dataset is multi-modal, meaning there are multiple modes. That is, the seven values above all appear equally frequently in the data.

Great. So far, so good. Let’s getting one more descriptive statistic, the standard deviation. There is a base R function for the standard deviation, sd(). So, let’s find out the standard deviation for arrests for murder in the USA:

sd(USArrests$Murder)

This should produce the following output:

Given that the standard deviation is quite large (it’s over half the size of the mean), it indicates a high degree of variability in the data. This suggests that the number of 6,arrests for murder in many states deviates significantly from the overall mean across the U.S. In other words, the murder arrest rates are not tightly clustered around the mean but are spread out, with some states having much lower or much higher rates than the national average.

# Understanding the Median in relation to the Mean

As the median is less than the mean, it suggests that the data is moderately right-skewed, suggesting the mean is being pulled up by more extreme high values. If the median was greater than the mean, it would mean the mean is being pulled down by extreme low values. Most states have arrest rates close to the median (7.250), but a few states with very high rates push the mean higher. This distribution highlights the inequality in murder arrest rates across states in 1973.
